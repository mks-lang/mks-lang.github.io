Why MKS

Low mystery. High visibility.

The whole pitch is simple: if the language has power, it should also have shape. You should be able to reason about what the runtime is doing.

GC

Transparent memory model

Mark/sweep is not a black box. The runtime is presented like a system you can observe, not a magic trick that quietly eats objects behind the curtain.

RX

Reactive kernel

watch and on change are built into the language surface, so reactive behavior feels native instead of bolted on with framework glue.

→

Sharp block syntax

Arrow blocks give the language a distinct silhouette. Control flow stays compact, readable, and weird in a good way.

Featured snippet

Watch + defer

runtime hooks 
watch x;
on change x -> Writeln("x changed", x); <-

defer -> Writeln("closing"); <-

var x =: 1;
x =: 2;

x changed 2
closing

Language feel

Not trying to be generic.

MKS gets stronger when it leans into its own identity: runtime visibility, reactive primitives, compact syntax, and system-flavored semantics.

Design goal

Readable power.

Useful features are great. Useful features that still look understandable on the page are better.
